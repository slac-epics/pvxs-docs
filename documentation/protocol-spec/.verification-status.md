# Protocol Specification Verification Status

This file tracks verification of every numbered subsection of the
three protocol specifications (`ca.rst`, `pva.rst`, `spva.rst`)
against authoritative reference materials. It is a working document
maintained alongside the specifications themselves; it is **not** part
of the published Sphinx site.

## Verification record format

For each numbered subsection: a row marked **agree**, **disagree**,
**partial**, or **unverifiable-by-this-source**. For disagreements,
the discrepancy is recorded with line citations on both sides; the
spec is then patched and the patch commit recorded.

#### PVA Section 5.1.1 — Size encoding

**Status:** ❌ **disagree** → patched in [TBD commit hash] of this same session.

| Source | 0–253 | 254 | 255 |
|---|---|---|---|
| Spec **as written** | 1-byte direct | 32-bit form follows | 64-bit form follows |
| `pvxs/src/pvaproto.h:265-301` | 1-byte direct | 32-bit form follows | nullable-null sentinel (`allow_null` only); otherwise protocol error |
| `cashark-pva` lines 993-1019 | 1-byte direct | 32-bit form follows | nullable-null sentinel (`is_nullable` only); otherwise protocol error |

**Spec was wrong.** The "64-bit form" does not exist. The byte 0xFF is a nullable-null sentinel (yields decoded value -1 in pva.lua / "null" semantically) for sizes appearing in nullable contexts (notably string fields that can be absent vs empty); in non-nullable contexts it is a protocol error.

**Patch:** Section 5.1.1 rewritten to remove the 64-bit form and add the nullable-null behavior.

#### PVA Section 5.5 (new) — BitSet Encoding

**Status:** ✅ **agree** with cashark-pva and pvxs (newly-authored section, source-grounded).

This section did not exist in the prior draft. Authored from `cashark-pva` lines 720–866 and 1020–1040, plus the decoder logic in lines 2240–2432. Cross-checks:

| Spec claim | Source confirmation |
|---|---|
| Wire encoding: Size n + n octets of bit data. | `cashark-pva:1029-1040` (`getBitSet`) |
| Bit-to-octet mapping: i/8 octet, i%8 bit, LSB-first within octet. | `cashark-pva:929-943` (`bufToBinary`); LSB-first explicit at line 938 ("LSB‑first"). |
| Bit-to-octet mapping does not depend on per-message MSB flag. | `cashark-pva:935` reads bytes in natural wire order; bit-shift inside the byte does not consult `is_big_endian`. |
| Depth-first pre-order indexing, root = bit 0. | `cashark-pva:744-777` (`walk` function); `start = index` initial, child indices = `start + 1` then increment by sub-tree size for each subsequent child. |
| Subtree size: leaf = 1; complex = 1 + sum(children). | `cashark-pva:792-802` (`sizeOf`). |
| Parent cascade: parent set AND no children set AND complex AND not union AND not any → mark all direct children. | `cashark-pva:760-769` (cascade conditional). |
| Cascade applies recursively. | `cashark-pva:771-774` (Pass 2 recurses depth-first into children after marking them; subsequent walks see them as set and apply cascade if applicable). |
| Cascade does NOT apply to union or any. | `cashark-pva:763-764` explicitly guards. |
| Structure-array gets a single bit (not per-element bits). | Inferred from `sizeOf` not recursing into structure-arrays at the per-element level (the array itself is a single child of its parent, with subtree-size 1 + children of the element-type description, where "element-type description" is itself a single FieldDesc node not a per-array-position expansion). Verified against pvxs `dataimpl.h` element-array handling (TODO: cite specific lines once dual-read of pvxs FieldDesc is done). |

**Patch:** new Section 5.5 added between former 5.4 and former 5.5. Former 5.5 (Type ID Lifetime) and former 5.6 (Normalised Types) renumbered to 5.6 and 5.7.

### CA Section 4.3 — Extended Header size

**Status:** ❌ **disagree** → patched.

| Source | Extended-header size |
|---|---|
| Spec **as written** | 32 octets (16 caHdr + 8 size+count + 8 reserved) |
| `cashark-ca:382-389` (`decodeheader`) | 24 octets (`hlen=24`) |
| `epics-base/modules/ca/src/client/cac.cpp:203` | `sizeof(caHdr) + 2 * sizeof(ca_uint32_t)` = 16 + 8 = 24 |
| `epics-base/modules/ca/src/client/tcpiiu.cpp:1168-1179` | When `m_postsize == 0xffff`: pop two more uint32 from queue (postsize_big and count_big). No additional 8-byte reserved field. |

**Spec was wrong.** There is no 8-byte reserved trailer; the extended header is exactly 24 octets total. The "32-octet" figure was fabrication on my part.

**Patch:** Section 4.3 table updated to remove the reserved row; section heading and prose reworded to "24-octet extended header" / "8-octet extended-header annex". Two subsection cross-references in Sections 1.2 and 3.4.3 also corrected.

### CA Section 6.2 — Positive Search Reply field meanings

**Status:** ❌ **disagree** (multiple field meanings) → patched.

| Field | Spec **as written** | `epics-base/modules/ca/src/client/udpiiu.cpp:683-712` says |
|---|---|---|
| `m_dataType` | server TCP port | server TCP port ✓ |
| `m_count` | server minor revision | reserved (server minor version is in payload, not in m_count) |
| `m_cid` | "server-chosen 32-bit (echoed back)" | pre-V4.8: server's IPv4 address. V4.8+: `INADDR_BROADCAST` sentinel; server IP comes from UDP source address. |
| `m_available` | client CID from request | client CID from request ✓ |
| Payload | "(V4.9+ only): client minor revision (V4.11+)" | V4.1+: 8 octets, bytes 0..1 are server minor revision in network byte order. NOT V4.9+, and NOT client revision. |

**Spec was wrong on three of five fields.**

**Note on dissector:** `cashark-ca:520-522` (`casearch`) reads the version at `buf(hlen+2, 2)` — bytes 2..3 of payload, not bytes 0..1. This appears to be a discrepancy between the dissector and `epics-base`. The spec follows the source (and an actual wire trace would resolve which one is correct in real traffic). Recorded here for follow-up; spec aligned with source for now.

### CA Sections 8.1.2, 8.2.2, 8.3.3 — m_cid in response messages carries STATUS, not SID

**Status:** ❌ **disagree** (three response-table corrections) → patched.

| Source | What `m_cid` carries in response messages |
|---|---|
| Spec **as written** (3 sections) | "server SID" |
| `epics-base/modules/database/src/ioc/rsrv/camessage.c:548-579` | "The m_cid field in the protocol header is abused to carry the status" — explicit comment. `cas_set_header_cid(pClient, ECA_xxx)` on error paths; success path leaves it at the default `ECA_NORMAL`. |
| `cashark-ca:631, 645, 687` | `t:add(feca, buf(8,4))` — reads offset 8 (m_cid slot) as ECA status code in all three response decoders. |

**Spec was wrong in three places** for ``CA_PROTO_READ_NOTIFY``, ``CA_PROTO_WRITE_NOTIFY``, and ``CA_PROTO_EVENT_ADD`` response/update messages. The `m_cid` field carries the CA status code, not the channel SID. The client correlates responses to requests via `m_available` (the IOI), not `m_cid`.

**Patch:** All three response tables (Sections 8.1.2, 8.2.2, 8.3.3) rewritten to make explicit that `m_cid` is the CA status code in these response messages, with cross-reference to Section 12.2. Surrounding prose updated to reflect that responses carry status (not SID) in the `m_cid` slot.

### PVA Section 3.2 — `EPICS_PVAS_*` vs `EPICS_PVA_*` precedence and applicability

**Status:** ❌ **disagree** (asymmetric behaviour between client and server, and between the two port variables) → patched.

| Source | Server `EPICS_PVAS_SERVER_PORT` / `EPICS_PVA_SERVER_PORT` | Server `EPICS_PVAS_BROADCAST_PORT` / `EPICS_PVA_BROADCAST_PORT` | Client `EPICS_PVAS_SERVER_PORT` / `EPICS_PVA_SERVER_PORT` | Client `EPICS_PVAS_BROADCAST_PORT` / `EPICS_PVA_BROADCAST_PORT` |
|---|---|---|---|---|
| Spec **as written** | "prefers `EPICS_PVAS_*` for server config" | (implied same: prefers `PVAS_*`) | "treats either name as equivalent for client config" | (implied same: treats either) |
| `pvxs/src/config.cpp:495` (server) | `pickone({"EPICS_PVAS_SERVER_PORT", "EPICS_PVA_SERVER_PORT"})` — both loaded, `PVAS_*` preferred when set | `pickone({"EPICS_PVAS_BROADCAST_PORT", "EPICS_PVA_BROADCAST_PORT"})` — both loaded, `PVAS_*` preferred when set | (not applicable on server side) | (not applicable on server side) |
| `pvxs/src/config.cpp:736` (client) | `pickone({"EPICS_PVA_SERVER_PORT", "EPICS_PVAS_SERVER_PORT"})` — both loaded, `PVA_*` preferred when set (note order reversed from server) | (not loaded) | (not loaded) | `pickone({"EPICS_PVA_BROADCAST_PORT"})` — ONLY this name; the `PVAS_*` variant is NOT loaded by clients |

**Spec was wrong on two counts:**

1. The spec said clients "treat either name as equivalent for client config" — true for `EPICS_PVA_SERVER_PORT` (clients do load both names with `PVA_*` preferred), but **NOT** true for `EPICS_PVA_BROADCAST_PORT`. Clients deliberately ignore `EPICS_PVAS_BROADCAST_PORT` because that variant configures which UDP port the local server BINDS, not which port a remote server is listening on; a client connecting to a remote PVA network must read the unsuffixed `EPICS_PVA_BROADCAST_PORT`.
2. The spec said servers "prefer `EPICS_PVAS_*` for server config" — true generally, but the actual precedence rule is "both loaded, `PVAS_*` preferred when set" rather than "`PVAS_*` only".

**Patch:** Section 3.2 rewritten to:
- specify server SHOULD honour both forms with `EPICS_PVAS_*` precedence when both are set;
- specify client SHOULD honour `EPICS_PVA_*` and SHOULD NOT honour the `EPICS_PVAS_*` variant for variables whose meaning differs between client and server (`BROADCAST_PORT` is the canonical example);
- note the special case of `SERVER_PORT` where both forms are equivalent on both sides (a server binds it; a client connects to it) and the asymmetric precedence preference.

### PVA Section 5.4 — FieldDesc kind-byte bit-level encoding

**Status:** ✅ **agree** with all three implementations (newly authored from the implementations as the source of truth, since the spec previously deferred this section to "future amendment").

The kind byte's bit-field structure was previously described as deferred. It is now specified normatively in Sections 5.4.2 (bit-field layout), 5.4.3 (enumerated values), and 5.4.4 (per-kind body). Sourced from:

| Source | Lines/files |
|---|---|
| `cashark-pva` | Lines 240-551 (`TYPE_CODE_*`, `isBoolType`, `isIntType`, `isFloatType`, `isStringType`, `isComplexType`, `isVariableArrayType`, `isBoundedArrayType`, `isFixedArrayType`, `getIntLen`, `isIntSigned`, `getFloatLen`, `getComplexName`) |
| `pvxs/include/pvxs/data.h` | Lines 141-205 (`Kind` enum, `TypeCode::code_t` enum, `kind() / order() / size() / isunsigned() / isarray()` accessors) |
| `epics-base/modules/pvData/src/pv/pvIntrospect.h` | Line 461 (`enum ArraySizeType { variable, fixed, bounded }`) |

The three sources agree on the bit-field layout. Bit assignments:

- Bits 7..5 (mask `0xE0`): **kind** — `000` Boolean, `001` Integer, `010` Real, `011` String, `100` Compound. Other values reserved.
- Bits 4..3 (mask `0x18`): **array variant** — `00` not array, `01` variable-length, `10` bounded, `11` fixed.
- Bit 2 (mask `0x04`): **signedness** for Integer kind; reserved for other kinds.
- Bits 1..0 (mask `0x03`): **detail** — for Integer/Real, scalar width; for Compound, sub-kind (struct/union/any).

**Implementation divergence note** (informative): the pvxs code at `pvxs/src/dataencode.cpp:120` treats array-variant bit `0x10` (bounded) and `0x18` (fixed) as a fault, with the comment "fixed length is deprecated". cashark-pva and pvAccessCPP / pvData (`pvIntrospect.h:461`) implement all three variants. The spec specifies the protocol space (all three variants); pvxs's deprecation of bounded/fixed is an implementation choice that does not change the wire-protocol space conforming peers may emit.

**Patch:** Section 5.4 expanded from the previous skeleton (which had a placeholder noting deferment) into a full specification: 5.4.1 lead-byte sentinels, 5.4.2 kind-byte bit-field layout with five sub-tables, 5.4.3 enumerated kind-byte values for ease of reference, 5.4.4 per-kind body encoding (scalar / variable-length array / bounded array / fixed array / structure / union / any / array of compound), 5.4.5 type-ID cache rules. ~280 lines of normative content added.

### SPVA Section 10 — PVACMS exposes operational PVs (health, metrics, status, issuer, root) AND action RPCs, not RPCs only

**Status:** ❌ **disagree (PVACMS exposes both readable/monitorable PVs and RPCs)** → patched.

| Source | What it says |
|---|---|
| Spec **as written** (Section 1.4 PVACMS terminology entry; Section 10.1 PVACMS-as-a-PVA-server intro; Section 10.2 well-known PV table) | "PVACMS itself runs as a PVA server, exposing its functions as RPCs at well-known PV names" — and the namespace table listed 5 PVs of which 4 were RPC and 1 was monitor (the per-cert STATUS PV). |
| `pvxs-cms/src/pvacms/serverhandle.cpp:249-255` | Server hosts seven distinct PVs: `create_pv` (SharedPV with onRPC handler — RPC entry), `schedule_pv` (SharedPV — RPC), `health_pv` (SharedPV — readable/monitorable), `metrics_pv` (SharedPV — readable/monitorable), `root_pv` (SharedPV — readable), `issuer_pv` (SharedPV — readable), `status_pv` (WildcardPV — readable/monitorable per-certificate status). |
| `pvxs-cms/src/pvacms/pvacms.cpp:682-711` | `makeHealthValue()` builds `<prefix>:HEALTH` as `epics:nt/NTEnum:1.0` with "OK"/"Not OK" choices plus ancillary fields (`db_ok`, `ca_valid`, `uptime_secs`, `cert_count`, `cluster_members`, `last_check`). Explicit comment: published as Normative Type so generic PVA clients (Phoebus, archiver, alarm server) can monitor it through standard tooling. |
| `pvxs-cms/src/pvacms/pvacms.cpp:714-719` | `makeMetricsValue()` builds `<prefix>:METRICS` as `epics:nt/NTScalar:1.0` (active certificate count) plus ancillary counters/timings, all individually subscribable. |
| `pvxs-cms/src/pvacms/pvacms.cpp:4286-4288` | `--health-pv-prefix` command-line option, default `CERT:HEALTH`. |

**Spec was wrong.** The "RPCs at well-known PV names" framing missed half of PVACMS's PV surface — the operational PVs that monitoring clients (Phoebus, alarm server, archiver) read and subscribe to without ever invoking an RPC.

**Patch:**

- Section 1.4 PVACMS terminology entry updated: now says PVACMS exposes "some of its functions as remote procedure calls (RPCs)" and "other functions as conventional monitorable / readable PVs: per-certificate status (subscribable), service health, service metrics, issuer-cert info, and root-CA info".
- Section 10.1 rewritten to explicitly distinguish two categories of well-known PVs PVACMS exposes — operational (state PVs accessed with `CMD_GET` and `CMD_MONITOR`) and action (RPC entry points accessed with `CMD_RPC`).
- Section 10.2 PV namespace table split into two separate tables: "operational" lists `<prefix>:STATUS:<skid>:<serial>`, `<prefix>:HEALTH`, `<prefix>:METRICS`, `<prefix>:ISSUER:<skid>`, `<prefix>:ROOT:<skid>`; "action / RPC" lists `<prefix>:CREATE`, `<prefix>:SCHEDULE`, `<prefix>:APPROVE`, `<prefix>:REVOKE`. HEALTH is documented as NTEnum with the actual ancillary fields from `makeHealthValue()`. METRICS is documented as NTScalar with the actual structure from `makeMetricsValue()`. The wildcard nature of the STATUS PV is also called out.

### CA Section 8.3.4 — EVENT_CANCEL final-update format

**Status:** ⚠️ **partial** (omitted normative detail) → patched.

| Source | What confirms an EVENT_CANCEL |
|---|---|
| Spec **as written** | "The server replies with the same message form (acknowledgment)" — implied symmetric `CA_PROTO_EVENT_CANCEL` echo from server. |
| `cashark-ca:691` | When the server emits a `CA_PROTO_EVENT_ADD` (cmd 1) with `msglen == 0` and the original subscription ID in `m_available`, the dissector labels it "Event Final" — i.e. the cancellation acknowledgment is a zero-payload `EVENT_ADD`, NOT an `EVENT_CANCEL` echo. |

**Spec was incomplete.** The actual cancellation-acknowledgment is a zero-payload `EVENT_ADD` (cmd 1), distinguishable by the empty payload, not a same-form `EVENT_CANCEL` echo. Clients use this final update as the signal to stop expecting further updates.

**Patch:** Section 8.3.4 prose updated to specify the final-update format explicitly.
