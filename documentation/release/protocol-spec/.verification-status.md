#### PVA Section 5.1.1 — Size encoding

**Status:** ❌ **disagree** → patched in [TBD commit hash] of this same session.

| Source | 0–253 | 254 | 255 |
|---|---|---|---|
| Spec **as written** | 1-byte direct | 32-bit form follows | 64-bit form follows |
| `pvxs/src/pvaproto.h:265-301` | 1-byte direct | 32-bit form follows | nullable-null sentinel (`allow_null` only); otherwise protocol error |
| `cashark-pva` lines 993-1019 | 1-byte direct | 32-bit form follows | nullable-null sentinel (`is_nullable` only); otherwise protocol error |

**Spec was wrong.** The "64-bit form" does not exist. The byte 0xFF is a nullable-null sentinel (yields decoded value -1 in pva.lua / "null" semantically) for sizes appearing in nullable contexts (notably string fields that can be absent vs empty); in non-nullable contexts it is a protocol error.

**Patch:** Section 5.1.1 rewritten to remove the 64-bit form and add the nullable-null behavior.

#### PVA Section 5.5 (new) — BitSet Encoding

**Status:** ✅ **agree** with cashark-pva and pvxs (newly-authored section, source-grounded).

This section did not exist in the prior draft. Authored from `cashark-pva` lines 720–866 and 1020–1040, plus the decoder logic in lines 2240–2432. Cross-checks:

| Spec claim | Source confirmation |
|---|---|
| Wire encoding: Size n + n octets of bit data. | `cashark-pva:1029-1040` (`getBitSet`) |
| Bit-to-octet mapping: i/8 octet, i%8 bit, LSB-first within octet. | `cashark-pva:929-943` (`bufToBinary`); LSB-first explicit at line 938 ("LSB‑first"). |

### PVA Section 3.2 — `EPICS_PVAS_*` vs `EPICS_PVA_*` precedence and applicability

**Status:** ❌ **disagree** (asymmetric behaviour between client and server, and between the two port variables) → patched.

| Source | Server `EPICS_PVAS_SERVER_PORT` / `EPICS_PVA_SERVER_PORT` | Server `EPICS_PVAS_BROADCAST_PORT` / `EPICS_PVA_BROADCAST_PORT` | Client `EPICS_PVAS_SERVER_PORT` / `EPICS_PVA_SERVER_PORT` | Client `EPICS_PVAS_BROADCAST_PORT` / `EPICS_PVA_BROADCAST_PORT` |
|---|---|---|---|---|
| Spec **as written** | "prefers `EPICS_PVAS_*` for server config" | (implied same: prefers `PVAS_*`) | "treats either name as equivalent for client config" | (implied same: treats either) |
| `pvxs/src/config.cpp:495` (server) | `pickone({"EPICS_PVAS_SERVER_PORT", "EPICS_PVA_SERVER_PORT"})` — both loaded, `PVAS_*` preferred when set | `pickone({"EPICS_PVAS_BROADCAST_PORT", "EPICS_PVA_BROADCAST_PORT"})` — both loaded, `PVAS_*` preferred when set | (not applicable on server side) | (not applicable on server side) |
| `pvxs/src/config.cpp:736` (client) | `pickone({"EPICS_PVA_SERVER_PORT", "EPICS_PVAS_SERVER_PORT"})` — both loaded, `PVA_*` preferred when set (note order reversed from server) | (not loaded) | (not loaded) | `pickone({"EPICS_PVA_BROADCAST_PORT"})` — ONLY this name; the `PVAS_*` variant is NOT loaded by clients |

**Spec was wrong on two counts:**

1. The spec said clients "treat either name as equivalent for client config" — true for `EPICS_PVA_SERVER_PORT` (clients do load both names with `PVA_*` preferred), but **NOT** true for `EPICS_PVA_BROADCAST_PORT`. Clients deliberately ignore `EPICS_PVAS_BROADCAST_PORT` because that variant configures which UDP port the local server BINDS, not which port a remote server is listening on; a client connecting to a remote PVA network must read the unsuffixed `EPICS_PVA_BROADCAST_PORT`.
2. The spec said servers "prefer `EPICS_PVAS_*` for server config" — true generally, but the actual precedence rule is "both loaded, `PVAS_*` preferred when set" rather than "`PVAS_*` only".

**Patch:** Section 3.2 rewritten to:
- specify server SHOULD honour both forms with `EPICS_PVAS_*` precedence when both are set;
- specify client SHOULD honour `EPICS_PVA_*` and SHOULD NOT honour the `EPICS_PVAS_*` variant for variables whose meaning differs between client and server (`BROADCAST_PORT` is the canonical example);
- note the special case of `SERVER_PORT` where both forms are equivalent on both sides (a server binds it; a client connects to it) and the asymmetric precedence preference.

### PVA Section 5.4 — FieldDesc kind-byte bit-level encoding

**Status:** ✅ **agree** with all three implementations (newly authored from the implementations as the source of truth, since the spec previously deferred this section to "future amendment").

The kind byte's bit-field structure was previously described as deferred. It is now specified normatively in Sections 5.4.2 (bit-field layout), 5.4.3 (enumerated values), and 5.4.4 (per-kind body). Sourced from:

| Source | Lines/files |
|---|---|
| `cashark-pva` | Lines 240-551 (`TYPE_CODE_*`, `isBoolType`, `isIntType`, `isFloatType`, `isStringType`, `isComplexType`, `isVariableArrayType`, `isBoundedArrayType`, `isFixedArrayType`, `getIntLen`, `isIntSigned`, `getFloatLen`, `getComplexName`) |
| `pvxs/include/pvxs/data.h` | Lines 141-205 (`Kind` enum, `TypeCode::code_t` enum, `kind() / order() / size() / isunsigned() / isarray()` accessors) |
| `epics-base/modules/pvData/src/pv/pvIntrospect.h` | Line 461 (`enum ArraySizeType { variable, fixed, bounded }`) |

The three sources agree on the bit-field layout. Bit assignments:

- Bits 7..5 (mask `0xE0`): **kind** — `000` Boolean, `001` Integer, `010` Real, `011` String, `100` Compound. Other values reserved.
- Bits 4..3 (mask `0x18`): **array variant** — `00` not array, `01` variable-length, `10` bounded, `11` fixed.
- Bit 2 (mask `0x04`): **signedness** for Integer kind; reserved for other kinds.
- Bits 1..0 (mask `0x03`): **detail** — for Integer/Real, scalar width; for Compound, sub-kind (struct/union/any).

**Implementation divergence note** (informative): the pvxs code at `pvxs/src/dataencode.cpp:120` treats array-variant bit `0x10` (bounded) and `0x18` (fixed) as a fault, with the comment "fixed length is deprecated". cashark-pva and pvAccessCPP / pvData (`pvIntrospect.h:461`) implement all three variants. The spec specifies the protocol space (all three variants); pvxs's deprecation of bounded/fixed is an implementation choice that does not change the wire-protocol space conforming peers may emit.

**Patch:** Section 5.4 expanded from the previous skeleton (which had a placeholder noting deferment) into a full specification: 5.4.1 lead-byte sentinels, 5.4.2 kind-byte bit-field layout with five sub-tables, 5.4.3 enumerated kind-byte values for ease of reference, 5.4.4 per-kind body encoding (scalar / variable-length array / bounded array / fixed array / structure / union / any / array of compound), 5.4.5 type-ID cache rules. ~280 lines of normative content added.

### SPVA Sections 2.2 / 3.6 / 4.5 — SNI extension and SAN-as-hostname-match (Web-PKI assumptions imported wrongly)

**Status:** ❌ **disagree** (three places import Web-PKI / RFC 6125 hostname verification semantics that SPVA does not implement) → patched.

The previous spec text claimed that:

1. (Section 2.2 typical-exchange walk-through, step 4) — "ClientHello includes ... `server_name` extension naming the server's expected DNS or IP name (per RFC 6066)".
2. (Section 3.6 "Server Name Indication") — "A client SHALL send the TLS `server_name` extension identifying the server it expects to connect to. The server's certificate Subject Alternative Name MUST contain a matching DNS name or IP address (Section 4.5)."
3. (Section 4.5 "Subject Alternative Name") — "`dNSName`: a DNS host name. For server certificates, this MUST match the hostname the client uses to connect (the `server_name` extension in TLS ClientHello)."

User question that triggered the verification: "Where would the client get this from?" — exactly the right question, because the answer is "nowhere".

| Source | What it shows |
|---|---|
| `pvxs/src/clientconn.cpp:148-200` (`Connection::startConnecting()`) | The pvxs client TLS-connect path creates the SSL object via `SSL_new`, attaches it to a libevent `bufferevent_openssl_socket_new`, configures the OCSP-stapling status callback, and calls `bufferevent_socket_connect(peerAddr)`. **Never calls `SSL_set_tlsext_host_name`. Never calls `X509_VERIFY_PARAM_set1_host`. Never calls `SSL_set1_host`.** No SNI, no hostname-matching configured anywhere. |
| `pvxs/src/openssl.cpp` (TLS context setup) | `SSL_CTX_set_verify` configures chain validation only (`mode = SSL_VERIFY_PEER` plus the cert-status callback `ossl_verify`). No hostname-verification flags set on the verify-param. |
| `pvxs/src/openssl.cpp:1240-1280` (peer-credentials extraction) | After successful TLS handshake, the SAN extension is read from the peer certificate and exposed to the application as `PeerCredentials.san[]` (a vector of `SanEntry{type, value}` pairs). This is metadata for authorization-rule matching at the application layer — NOT a verification check at the TLS layer. |
| `pvxs-cms/src/common/certfactory.cpp:370-403` (CCR-driven SAN issuance) | PVACMS embeds requester-supplied `dNSName` and `iPAddress` SAN entries verbatim in issued certificates. There is no implication anywhere in pvxs-cms that these are used for connection-time hostname matching. |

The connection target is `(IP, port)` from the PVA search reply (PVA spec Section 7.2 — IP comes from the UDP source address of the reply). There is no DNS name anywhere in the connection-establishment flow that a client could place in SNI. The whole framing was Web-PKI / RFC 6125 thinking imported wrongly into a protocol that uses chain-of-trust + cert-status PV monitoring as its authenticity guarantees.

**What actually authenticates the server (corrected spec wording):**

1. Chain validation (RFC 5280) against the configured trust anchor.
2. Cert-status PV check (Section 7).
3. Optional OCSP stapling (Section 13).

**What SAN is for (corrected spec wording):** supplementary identity metadata, exposed to the application via `PeerCredentials.san[]`, available to match against access security configuration file (ACF) rules at the authorization layer (Section 11). Never consumed at TLS-handshake time.

**Patch:**

- Section 2.2 step 4 — removed the `server_name` extension from the ClientHello list; added an explicit statement that ClientHello does NOT carry SNI.
- Section 3.6 — renamed from "Server Name Indication (SNI)" to "Server Name Indication (SNI) and Hostname Verification". Body completely rewritten: SPVA does NOT use SNI and does NOT perform hostname-versus-certificate verification; the three actual authenticity mechanisms (chain, cert-status, optional OCSP staple) listed; SAN exposed at the application layer for ACF matching, NOT consumed at TLS-handshake time.
- Section 4.5 — rewritten. Subject DN is the primary identity. SAN is supplementary metadata: stored at issuance, exposed via peer-credentials structure at handshake completion, available for ACF rule matching. The previous "MUST match the hostname the client uses to connect" claim deleted.

### SPVA Section 3.7 — TLS session resumption is not used (the previous spec invented an unimplemented mechanism)

**Status:** ❌ **disagree** (the entire mechanism the section described is not implemented; the prescribed restart-handling rules are vacuous because there is no resumption to begin with) → patched.

The previous spec said:

> TLS 1.3 session tickets are PERMITTED but OPTIONAL. Implementations SHOULD support them for performance. Tickets MUST NOT be reused across server restarts; servers MUST issue fresh PSK/resumption secrets on each startup.

User question that triggered the verification: "Resuming on server restart? I don't think this is implemented at all. For monitors a disconnect at the PVA layer is a full disconnect at lower levels and the only reason it reconnects is that the PVA layer searches and connects again when the same PV comes back online. Its a new connection completely behind the scenes lower down."

| Source | What it shows |
|---|---|
| `pvxs/src/` (full grep for `SSL_CTX_set_session_cache_mode`, `SSL_set_session`, `SSL_get_session`, `SSL_CTX_set_num_tickets`, `SSL_OP_NO_TICKET`, `SSL_SESS_CACHE`, `session_ticket`, `session_resume`) | **Zero hits.** pvxs never configures the OpenSSL session subsystem, never caches sessions on the client, never installs ticket-key callbacks on the server, never calls `SSL_set_session` to attempt resumption. |
| `pvxs/src/clientconn.cpp:148-200` (`Connection::startConnecting()`) | Creates a fresh `SSL` object via `SSL_new` on every connect. No `SSL_set_session(ssl, cached)` call anywhere. |
| `pvxs/src/clientconn.cpp:425-479` (`Connection::cleanup()`) | On disconnect: `bev.reset()` tears down the bufferevent (and with it the underlying `SSL` and `TCP`); the connection is removed from `connByAddr`; every channel on the connection is returned to the SEARCHING state via `chan->disconnect(chan)`. No SSL session cached anywhere across this teardown. |
| `pvxs/src/openssl.cpp` (TLS context setup) | `SSL_CTX_set_verify` configures peer-cert verification only. No `SSL_CTX_set_session_cache_mode`, no `SSL_CTX_set_session_id_context`, no ticket-key callbacks. The OpenSSL defaults apply: TLS 1.3 servers emit `NewSessionTicket` messages, but clients (which never call `SSL_get1_session()` to pin them) discard them. |

The actual SPVA reconnection model (verified against pvxs):

1. Disconnect occurs (TCP RST/FIN, network error, cert-status transition to a non-good state, etc.).
2. `cleanup()` tears down the bufferevent (full TLS+TCP teardown), removes the connection from the address-keyed map, returns every affected channel to the SEARCHING state.
3. The PVA layer re-issues `CMD_SEARCH` UDP queries.
4. A search reply arrives (possibly from a different server IP, or the same server now holding a different certificate after restart).
5. Fresh TCP connect to the responding `(IP, port)`.
6. Fresh TLS handshake (full handshake — no resumption attempted).
7. Fresh `CMD_CONNECTION_VALIDATION` PVA-level handshake.
8. Channels re-create on the new connection via `CMD_CREATE_CHANNEL`.
9. Application-layer monitor subscriptions re-establish via `CMD_MONITOR/Init`.

There is no cross-connection TLS state preserved at any step. The "resumed from SUSPENDED" log message at `clientconn.cpp:384` is the unrelated pvxs-internal monitor pause/resume mechanism for `SUSPENDED` certificate-status state, in which the TLS connection is kept up while the certificate is temporarily out-of-good-state and monitors are paused-then-resumed without round-tripping; that is an in-connection pause/resume, not cross-connection TLS resumption.

**Patch:** Section 3.7 retitled "Session Resumption (Not Used)" and rewritten to state explicitly that SPVA does not use TLS 1.3 session resumption; clients MUST NOT cache or replay session tickets; the rationale (PVA's full-disconnect-and-fresh-search reconnection model) is given. The previous text's "fresh PSK/resumption secrets on each startup" prescription is removed entirely as describing an unimplemented mechanism.

### CA Section 8.3.4 — EVENT_CANCEL final-update format

**Status:** ⚠️ **partial** (omitted normative detail) → patched.

| Source | What confirms an EVENT_CANCEL |
|---|---|
| Spec **as written** | "The server replies with the same message form (acknowledgment)" — implied symmetric `CA_PROTO_EVENT_CANCEL` echo from server. |
| `cashark-ca:691` | When the server emits a `CA_PROTO_EVENT_ADD` (cmd 1) with `msglen == 0` and the original subscription ID in `m_available`, the dissector labels it "Event Final" — i.e. the cancellation acknowledgment is a zero-payload `EVENT_ADD`, NOT an `EVENT_CANCEL` echo. |
